# 面试 static

问：什么时候会使用static？ 当一个方法或变量需要初始化的时候，或者经常被掉用的时候可以加上static; static 可以修饰变量，或方法或类； 修饰的变量为静态成员变量，类加载的时候被分配堆上创建，可以被整个类访问，可以直接类名直接访问，也可以使用对象名访问； 修饰的方法为静态方法，只能访问本类的静态成员变量，访问非静态成员通过创建类的对象，通过对象访问； 普通方法，可以直接访问同类中的额非静态变量和静态变量； 静态方法中，不能直接调用非静态方法，但是可以通过对象来访问非静态方法；

# 类加载机制是什么？

javac编译器会将.java文件编译成.class文件，当java虚拟机需要某个类的时候会将.class文件加载并创建类的对象，这个过程称为类的加载过程； 类加载过程分为 加载（由classloader根据类的全限定名加载class文件） 验证（验证文件信息如文件格式的验证，元信息验证，字节码验证，符号引用验证） 准备（为类变量 staic修饰的字段变量 分配空间并赋初始值，不包含final修饰的static,final在编译的时候已经分配了，不会为实例变量分配初始化） 解析（把常量池的符号引用替换成直接引用） 初始化（如果该类具有父类就对父类进行初始化，执行静态代码块和静态初始化成员变量）

# 问http与https的区别，s指的什么？

http是超文本传输协议，负责客户端到服务端的一系列操作，专门用来注入html的超媒体文档等web内容的协议，基于传输层的tcp协议的应用层协议。 https:是基于安全套接字的http协议，http+ssl/tls（数字证书） 区别：http不安全，https安全 标准端口：80 443 https传输层加密 https工作原理： 首先服务器端向客户端发送传输证书，这个证书是公钥，包含了如证书的办法机构，过期时间等； 客户端收到公钥会解析验证证书，如果没有问题，生成一个随机值（私钥）然后用证书对这个私钥加密，发送给服务端； 服务端使用私钥对信息解密，得到客户端的私钥，然后客户端和服务端就通过这个私钥通信 （非对称的加密传输协议）

# 问聊一下mysql的索引？

索引分为单列索引（单个列的索引，一张表可以有很多单列索引），唯一索引（索引列的值必须唯一，但允许空值），聚合索引（多个列的索引） Btree索引，hash索引（hash索引是在btree索引基础上对索引进行hashCode）

# get和post的区别

1：传送方式：get是浏览器传输，post是报文传输 传送长度：get有参数限制（取决于浏览器），post上理论上没有限制 2，数据传输：get，通过拼接url进行传递参数，url是可见； post，通过body体传输参数，url不可见 3，GET产生一个TCP数据包，浏览器会把http header和data一并发送出去，服务器响应200(返回数据); POST产生两个TCP数据包，浏览器先发送header，服务器响应100 continue，浏览器再发送data，服务器响应200 ok(返回数据)。 4，缓存方式：GET请求会被浏览器主动缓存，而POST不会，除非手动设置。

# mysql优化。

①：联合索引第一个字段使用范围不会走索引； ②：覆盖索引优化； ③：in和or在比表数据量比较大的情况会走索引，在表记录不多的情况会选择表扫描； ④：like kk%一般情况都会走索引，尽量采用后置的通配符，用到了索引下推优化，mysql5.6引入了索引下推优化，可以在索引遍历中个，对索引中包含的所有字段先做判断，过滤掉不符合条件的记录之后再回表，到主键索引上找出相应的记录； 索引下推会减少回表次数，对于innodb引擎的表索引下推只能用于二级索引，innodb的主键索引（聚簇索引）树叶子节点上保存的是全行数据，所以索引下推并不会起到减少查询全行数据的效果 ⑤：尽力避免NULL值出现，可能会造成索引失效； ⑥：查询条件上尽量不要对索隐裂使用函数； ⑦：<>,not, in ,not exists 条件可能会倾向于全表扫描

# 数组和链表的数据结构的差异

LinkedList节点查找： LinkedList 并没有采用从头循环到尾的做法，而是采取了简单二分法，首先看看 index 是在链表的前半部分，还是后半部分。 如果是前半部分，就从头开始寻找，反之亦然。通过这种方式，使循环的次数至少降低了一半，提高了查找的性能。

LinkedList的获取元素操作实现跟LinkedList的删除元素操作基本类似，通过分前后半段来循环查找到对应的元素，但是通过这种方式来查询元素是非常低效的，特别是在for循环遍历的情况下，每一次循环都会去遍历半个List。 所以在LinkedList循环遍历时，我们可以使用iterator方式迭代循环，直接拿到我们的元素，而不需要通过循环查找List。 所以不推荐使用for循环，应使用迭代器

public class RedisTool {


    private static final String LOCK_SUCCESS = "OK";
    private static final String SET_IF_NOT_EXIST = "NX";
    private static final String SET_WITH_EXPIRE_TIME = "PX";


    /**
     * 尝试获取分布式锁
     * @param jedis Redis客户端
     * @param lockKey 锁
     * @param requestId 请求标识
     * @param expireTime 超期时间
     * @return 是否获取成功
     */
    public static boolean tryGetDistributedLock(Jedis jedis, String lockKey, String requestId, int expireTime) {


        String result = jedis.set(lockKey, requestId, SET_IF_NOT_EXIST, SET_WITH_EXPIRE_TIME, expireTime);


        if (LOCK_SUCCESS.equals(result)) {
            return true;
        }
        return false;
    }
}

    • 哨兵至少三台，作用有(集群状态监控、消息通知、主从切换、配置中心)
    • 哨兵也是分布式部署的

    • 先启动主节点，再从节点，再哨兵。

      • 从节点先配置主从。

      • 主从节点配置哨兵

      • 哨兵配置文件(sentinel.conf)

        • sentinel monitor

          • matser节点名称 ip 端口号
          • count指存在count台哨兵认为master节点死亡时才会主从切换。
          • count不得大于哨兵集群的数量

          • 超时时间，单位毫秒。多久内连接不上判定节点不可用，状态异常

        • sentinel parallel-syncs 1

        • sentinel auth-pass

          • 如果主节点配置有密码，需要在哨兵节点添加密码。

    • 最少3master，三主三从。
    • 共享16384槽位，CRC16算法确定槽位。

spring事务的实现原理：

spring事务传播机制：

  • REQUIRED(必须有事务,没有事务则新建)、REQUIRES_NEW(不支持当前事务，新建事务)、NESTED、SUPPORTS(可有可无，有事务就事务执行，否则非事务执行)、NOT_SUPPORTED(非事务执行，存在事务则挂起)、 MANDATORY(强制事务，没有则抛出异常)

  • 事务的传播特性指的是不同方法嵌套调用过程中，事务如何处理(规则)。是使用同一个事务还是不同事务，出现异常是回滚还是提交。日常使用中较为频繁的REQUIRED、REQUIRES_NEW、NESTED(嵌套事务，使用外层方法事务)。

  • 事务传播等级可分为三类：支持当前事务、不支持当前事务、嵌套事务。

  • 核心处理逻辑：

    • 判断内外层是否为同一个事务

      • 是：异常统一在外层进行处理
      • 否：内层方法有可能影响外层，但外层不会影响内层

REQUIRED与NESTED回滚的区别：

  • REQUIRED与外层一起成功一起失败
  • NESTED内层方法异常不会影响外层提交

    • =主动删除+惰性删除
    • 每隔一段时间，主动删除。不主动删除时执行惰性删除。

  • 读写锁 redisson redisson的读写锁也是基于lua脚本实现的(适用于读多写少的场景)

  • redisson获取读写锁对象，读缓存的读操作加读锁，写操作加写锁。

    • 两进程一个读 一个写 不会存在互斥问题 两个都写 则序列化进行

threadLocal(线程局部变量) 作用及使用场景：

  • 让变量私有化，为每个线程提供变量的副本，以此实现线程变量的隔离，各自操作自己的变量副本，不受影响。 一般静态修饰，防止内存泄露 remove使用完毕后移除

    • 原理：内部存在一个threadLocalMap,key即为threadLocal map就是value。
    • 场景 多数据源配置切换

  6. 将第四步找到的线程编号转16进制，在dump(信息查找相应内容)
  7. 根据线程信息，定位到具体的代码

JVM优化原则

  • GC调优3选2(吞吐量、延迟、内存)
  • 不能为了优化而优化，上线时尽量把JVM参数设置到最优。
  • 先优化代码、架构，再考虑JVM优化。

什么情况下需要优化

  • 堆内存(老年代)持续上涨达到设置的最大内存值
  • fullGC频繁
  • GC停顿时间过长(超过1s)
  • 应用出现内存溢出异常
  • CPU或内存使用占比过高不下
  • 系统吞吐量及性能下降

  • 继承：从现有类派生新类，子类可看做特殊的父类。
  • 多态：类和类之间的关系，父类引用可指向子类对象。

Spring事务的原理：

  • 使用声明式事务时，事务由spring框架进行控制。
  • spring会基于此类(@Transactional修饰的)生成一个代理对象，然后在使用时如果有事务处理，会把事务的自动提交给关闭，先执行具体的业务逻辑，没问题就提交，有问题就回滚。

Spring事务的失效问题：

  • bean对象没有被容器管理

  • 方法非public

    • AOP没办法为这样的方法生成一个代理

  • 自身调用

  • 数据库不支持事务

  • 数据源没有配置事务管理器

  • 异常被捕获

  • 异常类型错误或配置错误

    • 抛出异常为配置异常的上层
    • 默认配置runtimeException

消息丢失

kafka

生产端->kafka结点消息丢失

  1. 设置ack登记
  2. 开启重试策略
  3. 使用带有回调的send()方法

kafka结点->消费者 消费者拉到消息未消费挂掉了

  1. offersets记录消费者消费的位置
  2. 设置auto.commit.enable=false，每次处理完手动提交

kafka结点丢失

  1. 配置多副本

rabbitMQ

生产者消息发送失败

  1. 生产者开启confirm(消息确认机制)

    1. 设置信道confirm模式，经该信道发送的消息有唯一的标识。

  3. 生产者事务方式发送消息

rabbitMQ接收消息后丢失

  1. 开启rabbitMQ的持久化。
  2. 发送消息时设置消息的deliveryMode为2，进行消息持久化

消费者消息丢失

  1. 关闭自动ack，消费后进行手动ack。

消息重复(重复消费)

Kafka

生产者消息重复 kafka节点响应异常，导致生产者重试。

  1. 开启kafka结点的幂等性 设置应答等级-1，存在副本且应答最小副本数大于2
  2. ack=0,不重试。 存在消息丢失的风险

消费者消息重复 消费后 宕机没有返回最新offset导致重复消费

消费者处理耗时超过max.poll.interval.ms(两次拉取时间的间隔)导致认为当前消费者已经死掉触发再平衡。

  1. 手动ack

  2. 消费者实现消费幂等性

    1. 缓存消费的消息id
    2. 数据库唯一索引等

  4. 针对于处理超时的情况，可以采用消息拉去后异步处理、减少每次拉取消息量、增加两次拉取间的处理间隔等方法进行优化。

rabbitMQ

  1. 生产者消息幂等性，确保不存在重复消息推送

  2. 消费端也实现幂等性，确保不重复消费

    1. 例如主键冲突之类的。
